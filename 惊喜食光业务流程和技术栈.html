余味盲盒 · 产品技术方案

临期不临味
拯救食物浪费

连接线下烘焙门店与价格敏感型消费者，将每日剩余库存以盲盒形式低价出售，实现商家清库存、消费者得实惠、社会减浪费的三方共赢

Detail Flow
核心业务详细流程
以"一个盲盒从创建到被消费"的完整生命周期为主线，拆解每一步的前置条件、系统动作和异常分支。

商家侧流程

1. 商家入驻申请

提交营业执照、食品经营许可证、门店照片等资质材料，平台人工审核（T+1）

2. 商品库配置

录入店内商品信息（名称、分类、原价、日均产量），支持批量导入

3. 打烊前盘点（16:00-17:00）

勾选当日剩余商品，系统自动计算原价总额，推荐折扣档位

4. 一键打包发布盲盒

设置盲盒名称、售价、份数、取货时间窗，发布后即时对用户可见

5. 接收订单 / 备货打包

用户下单后收到通知，提前将盲盒打包好，贴上取货码标签

6. 到店核销

用户到店出示取货码，商家扫码确认核销，系统状态更新为"已完成"

7. 超时处理

超过取货时间窗未核销 → 系统自动退款 → 商家可选择继续降价或放弃

用户侧流程

1. 注册授权

微信一键登录，获取手机号，完善昵称头像（非必填）

2. 发现盲盒

首页推荐 + LBS附近门店 + 分类筛选 + 排序（距离/价格/折扣），浏览盲盒列表

3. 查看详情

查看盲盒说明、门店信息、取货时间窗、剩余份数、折扣力度

4. 下单支付

微信支付，支持优惠券抵扣，支付成功后生成取货码（4位数字）

5. 到店取货

在取货时间窗内到店 → 出示取货码 → 商家扫码核销 → 拿到盲盒

6. 开箱体验

拆开盲盒，获得随机组合的当日剩余烘焙产品

7. 评价分享（可选）

对本次购买打分评价，可分享至好友/朋友圈，获得环保积分奖励

异常分支与兜底策略

用户超时未取
取货时间窗结束后 → 系统自动触发退款 → 通知商家 → 商家可选择：重新降价发布 / 直接废弃 → 记录超时率影响用户信用分
商品质量投诉
用户发起投诉 → 平台客服介入 → 要求用户提供照片证据 → 判定责任方 → 如商家责任则全额退款 + 扣商家分 → 如恶意投诉标记用户
盲盒售罄后下单
使用乐观锁控制库存 → 并发下单时只有一人成功 → 其余提示"手慢了" → 推荐 similar 盲盒 → 降低用户流失

小程序 uni-app (Vue3)3.4+ 一套代码同时编译微信小程序和H5，Vue3 Composition API 开发效率高，插件生态丰富，未来可扩展至App端
微信原生 (备选)- 如对性能要求极高或需要调用大量原生能力时切换，初期不建议，开发成本高
管理后台 Vue 3 + Vite3.4+ Composition API + setup语法糖，Vite构建速度快，TypeScript支持好，适合中后台复杂表单和表格场景
Element Plus2.5+ Vue3最成熟的企业级UI组件库，表格/表单/弹窗等中后台高频组件完备，文档中文友好
后端框架 Spring Boot 33.2+ Java生态最成熟的Web框架，事务管理/安全框架/ORM集成完善，团队招聘容易，适合支付/订单等对事务一致性要求高的业务
微服务 Spring Cloud Alibaba2022.x Nacos注册中心+配置中心，Sentinel限流熔断，Gateway网关，国内文档和社区活跃度最优
关系数据库 MySQL8.0+ 订单/用户/商家/盲盒等结构化数据，InnoDB事务支持，主从复制成熟，云厂商RDS托管方便
缓存 Redis7.x 盲盒库存扣减（分布式锁）、取货码生成与校验、LBS附近门店缓存、热点盲盒列表缓存、用户Session
搜索引擎 Elasticsearch8.x 盲盒全文搜索+多条件筛选、门店基于geo_distance的LBS附近查询，性能远优于MySQL LIKE
支付 微信支付 v3API v3 小程序内支付的唯一选择，JSAPI下单+回调通知，需处理重复通知、签名验证、退款等边界情况
对象存储 腾讯云 COS- 商家营业执照/食品许可证/门店照片/盲盒图片，CDN加速分发，按量付费成本低

System Relations
四端系统职责边界与交互关系
明确每个系统"管什么、不管什么"，以及系统之间通过什么接口通信。
C端

用户端小程序

核心职责：浏览盲盒、下单支付、取货码展示、订单管理、个人中心、环保成就

不管什么：不管商家审核、不管结算、不管数据统计

调用接口：盲盒列表(ES)、盲盒详情、创建订单、微信支付、订单查询、核销状态查询、用户信息CRUD

B端

商家端小程序

核心职责：库存盘点、盲盒创建发布、订单通知、扫码核销、日结数据查看

不管什么：不管用户管理、不管跨店数据、不管平台配置

调用接口：商品库CRUD、盲盒创建/上架/下架、订单列表(本店)、扫码核销、商家数据看板

运营

平台管理后台

核心职责：商家入驻审核、用户管理(封禁/解封)、全量订单查看、内容运营、财务结算、系统配置、数据看板

权限模型：RBAC角色权限，超级管理员/商家运营/用户运营/客服，菜单级+按钮级权限控制

调用接口：全部API均可访问，通过角色权限过滤

商家PC

商家管理后台

核心职责：小程序功能的PC端增强版，侧重数据分析和批量操作：多维度销售报表、商品库批量管理、财务提现、店铺设置

与小程序差异：小程序侧重移动端轻操作(盘点/核销)，PC端侧重重分析(报表/配置)，数据互通但交互形态不同

调用接口：与商家小程序共享同一套商家API，通过merchant_id隔离数据

核心原则
🏗️
一套后端服务
四端共享同一套Spring Cloud微服务集群，通过不同的API前缀和角色权限区分调用方
🔐
数据隔离靠权限
商家端所有接口带merchant_id过滤，用户端带user_id过滤，平台端可看全量。同一张表，不同视角
🔄
状态驱动流转
订单状态机（待支付→待取货→已完成/已退款/已超时）是核心枢纽，四端围绕状态变化做各自的事
